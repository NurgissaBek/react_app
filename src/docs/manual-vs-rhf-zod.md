1. Какие части ручной реализации заменил `useForm`?
   Хранение данных (теперь через useform а не usestate,это хорошо потому что при usestate каждый раз когда что то мелкое менялось то происходил ререндер формы и в случае с сложной формой она начала бы лагать, RHF хранит всё в обычном объекте через useRef, а изменение ref ререндер не вызывает), инициализацию (createInitialValues просто забивала поля пустыми строками при старте. Если бы позже нужно было бы сбросить форму (например, для кнопки «Очистить»), пришлось бы вручную обнулять кучу переменных. Теперь один раз передаем начальные значения в defaultValues. Библиотека запоминает эту «точку отсчета» и позволяет сбросить и текст в полях, и ошибки, и статусы кнопок одной короткой командой reset().), отслеживание ввода (вместо одного самописного handleChange, который по event.target.name определял поле, персональный onChange из register для каждого поля), поведение при ошибках (ручной код при вводе просто стирал текст ошибки, даже если поле оставалось невалидным, а reValidateMode: 'onChange' перепроверяет по-настоящему — ошибка уходит, только когда поле реально исправлено; и фокус на первое невалидное поле, который делался руками через массив FIELD_NAMES, теперь делает shouldFocusError.)

2. Что заменил `zodResolver`?
   валидацию, связку из функции validateForm, словаря правил и конфигурации полей. По сути, это мост-переводчик между React Hook Form и библиотекой Zod. Было - validateForm перебирала поля, проверяла их через регулярные выражения или условия (пример-minDigits: 2 и тд) и вручную формировала объект с ошибками errors. стало - Вся валидация описана в одной Zod-схеме, а резолвер связывает её с формой.При отправке формы zodResolver забирает значения из хранилища useForm и передает их в Zod-схему. Затем он переводит результат проверки на язык библиотеки и если нет ошибок то передает данные в onSubmit если же есть то записывает его в formState.errors после чего они отоброжаются на экране

3. В чём теперь находится источник истины для значений формы?
   DOM(браузер) , при вызове register() библиотека сохраняет прямую ссылку на DOM-узел инпута в хук useRef. Когда данные отправляются (onSubmit) или перепроверяются, RHF не ищет их в стейте, а обращается напрямую к свойству DOM-элемента — ref.current.value.

4. Чем `register` отличается от ручных `value` и `onChange`?
   value + onChange гоняла каждую букву через React: клавиша -> setState -> перерисовка страницы -> значение вернулось в поле. С register буквы показывает сам браузер, RHF просто слушает и запоминает — перерисовок при печати ноль.

5. Почему React Hook Form часто называют подходом с неконтролируемыми полями?
   потому что библиотека полностью убирает проп value из инпутов, передавая управление текстом самому браузеру (DOM). В контролируемом подходе React жестко диктует инпуту, какую букву показывать, обновляя стейт на каждый символ, из-за чего форма постоянно перерисовывается. В RHF за вывод букв на экран отвечает сам DOM-элемент, а библиотека лишь подключается к нему через ссылку (ref) для чтения значений и подслушивает изменения. Так как React больше не навязывает инпутам их содержимое через value и не контролирует каждый шаг пользователя, этот подход и называется неконтролируемым.

7. Почему преобразование строк из `<input type="number">` требует отдельного внимания?
   потому что любой HTML-инпут всегда возвращает текстовую строку даже если у него указан числовой тип. Главная проблема кроется в поведении JavaScript где приведение пустой строки к числу (Number('')) дает результат 0. Если преобразовывать данные бездумно, то пустое незаполненное пользователем поле формы автоматически превратится в ноль. В результате валидация на обязательность поля (Поле обязательно для заполнения) молча сломается, так как система воспримет этот ноль как реальное, валидное значение (например, 0 км пробега).

8. Чем Zod-схема отличается от прежнего `validatorConfig`?
   Прежний конфиг был просто набором условий который проверял поле на валидность но оставлял данные как есть — в виде сырых строк. Zod-схема работает как полноценный контракт: она не только проверяет данные но и одновременно трансформирует их. Благодаря этому onSubmit получает полностью готовый, чистый объект, где строки очищены от пробелов, селекты строго соответствуют константам, а инпуты с пробегом гарантированно стали числами.

9. Что стало проще при добавлении нового поля?
   Процесс сократился до двух шагов вместо четырех: теперь достаточно добавить одну строку в Zod-схему и одну строку в JSX, вызвав метод register. Раньше для этого приходилось проделывать кучу ручной работы: обязательно вносить имя поля в массив FIELD_NAMES, прописывать для него отдельный блок правил в конфиге валидации, при необходимости создавать новые условия в общем словаре и вручную навешивать на инпут три разных пропа.

10. Что стало сложнее или менее очевидно?
    Сложнее стала отладка кода. Когда вся логика умещалась в 60 строк собственного кода, её можно было легко прочитать сверху вниз и сразу найти ошибку. Теперь для поиска багов нужно четко понимать внутреннюю кухню двух сторонних библиотек: как именно React Hook Form управляет ререндерами и в каком порядке Zod запускает свой конвейер проверок.

11. Какие зависимости и привязку к библиотекам мы добавили?
    три новых пакета: react-hook-form, zod и @hookform/resolvers

12. В каких случаях простая ручная форма всё ещё может быть разумнее?
    Ручной подход оправдан для мини-форм на 1–3 поля с простейшей логикой вроде строки поиска, авторизации или подписки на рассылку — в таких случаях подключение тяжелых библиотек принесет больше лишнего веса, чем реальной пользы. также на этапе обучения чтобы в целом понять как работает форма и какую рутинную работу берут на себя библиотеки
